## Mandatory Requirements

### Proper English Requirement (NON-NEGOTIABLE)

All AI assistants MUST use proper English in every response.

Required standards:
- Use correct grammar, spelling, and punctuation.
- Use clear, professional, and complete sentences.
- Avoid slang, shorthand, and informal abbreviations (unless quoting user input).
- Use consistent technical terminology across code, documentation, and chat responses.
- If user input contains grammar mistakes, respond in proper English while preserving user intent.

Scope:
- This rule applies to chat responses, code comments, documentation, and commit messages generated by AI assistants.

Compliance:
- Output that does not meet proper English standards is non-compliant.

### English QA Checklist (Required Before Sending)

- [ ] Grammar is correct.
- [ ] Spelling is correct.
- [ ] Punctuation and capitalization are correct.
- [ ] Wording is clear, concise, and professional.
- [ ] Technical terms are used consistently.

### 1. Always Reference Standards

When responding to user requests:
- Always cite the relevant standard or guideline.
- Always explain how the proposed solution aligns with standards.
- Always validate against `docs/standards/golang_best_practices.md`.

### 2. Code Quality Standards

Every code change must meet these criteria:
- Must pass `go fmt` and `go vet`.
- Follow Go naming conventions.
- Handle errors with context.
- Add/update tests for behavior changes.
- Document exported symbols with godoc.

### 3. Project Structure Compliance

Always follow the recommended project structure and keep related functionality together.

### 4. Error Handling Pattern

Every error scenario must include immediate checks and wrapped context.

### 5. Documentation Pattern

Every exported symbol must have clear godoc comments.

### 6. Dependency and License Compliance (NON-NEGOTIABLE)

All dependencies must be permissive and approved.

Allowed:
- MIT
- Apache 2.0
- BSD (2/3 clause)
- ISC
- Unlicense / CC0 (when appropriate)

Forbidden:
- GPL / AGPL / LGPL / SSPL
- Proprietary or unknown licenses

### 7. Performance and Reliability Validation (NON-NEGOTIABLE)

AI assistants MUST evaluate performance and reliability where applicable.

Required standards:
- **Benchmarks:** Add and run benchmarks only if the user explicitly asks for benchmarks.
- **Coverage:** Run and report coverage only if the user explicitly asks for coverage.
- **Deadlocks:** Always review concurrency code for potential deadlocks.
- **Memory leaks:** Always review long-lived processes and high-allocation paths for memory leaks.

Deadlock checks:
- Inspect mutex/channel/waitgroup usage for lock-ordering issues, blocked sends/receives, and goroutine lifecycle leaks.
- Validate context cancellation and goroutine shutdown behavior.
- Flag any potential deadlock risk explicitly, even if unconfirmed.

Memory leak checks:
- Inspect repeated allocations and long-lived references.
- Verify goroutines/tickers/timers/resources are stopped or released.
- Use profiling/tools where applicable to investigate suspicious growth patterns.

When explicitly requested by the user:
- **Benchmark commands:**
   - `go test -bench=. -benchmem ./...`
   - `go test -run=^$ -bench=. -benchtime=3s ./...`
- **Coverage command:**
   - `go test -cover ./...`

Always-run baseline validation:
- `go test ./...`
- `go test -race ./...`

Response requirements:
- If benchmarks/coverage were requested, include results.
- Always state deadlock and memory-leak review outcome explicitly.
- If a check is not applicable, explain why.
